# MCP Server Requirements & Target Integration

When deploying Model Context Protocol (MCP) servers as targets behind the Bedrock AgentCore Gateway, you must configure your servers to correctly handle request transports, multi-tenant authentication, and AWS credential injection.

## Transport Requirements

The AgentCore Gateway acts as a remote proxy and requires the following to connect to your MCP tools:

- **Streamable HTTP (Required):** The Gateway communicates via `HTTP POST` using the `streamable-http` MCP transport. 
- **No STDIO Support:** Because the Gateway is a cloud service routing to remote endpoints, it cannot spawn your MCP servers as local child processes via `stdio`.
- **Stateless HTTP (Default):** AgentCore injects an `Mcp-Session-Id` header to manage sessions, meaning your servers natively start in stateless mode. (Stateful HTTP is required only if you are using advanced sampling or multi-turn elicitation features).

## Credential Injection Flow

To enforce true Just-in-Time (JIT) access, the MCP server *must* use the temporary AWS STS credentials generated by the agent's current session, not the global credentials of the server hosting it.

1. **The Interceptor Extraction:** When a request is sent to the Gateway, our `Lambda Interceptor` reads the Workload Token, `x-target-account-id`, and `x-target-role-name`.
2. **Credential Generation:** The Interceptor triggers `GetResourceCredentials` against AWS IAM Identity Center (IDC) to generate short-lived AWS STS Access Keys mapped to the user's specific Entra ID session.
3. **The Header Injection:** The Interceptor transparently injects these STS credentials into the HTTP request bound for your MCP target:
   - `x-aws-access-key-id`
   - `x-aws-secret-access-key`
   - `x-aws-session-token`

### How Your MCP Server Needs to React
For the MCP server (e.g., `aws-api-mcp-server`) to securely execute actions in the specific target account, **it must extract these three `x-aws-*` headers** from the incoming HTTP request and instantiate a localized `boto3` client specific to that specific tool execution. 

## Hosting Options (Handling Multi-Tenancy)

Many off-the-shelf open-source MCP servers (such as [`awslabs/mcp`](https://github.com/awslabs/mcp/tree/main/src/aws-api-mcp-server) `aws-api-mcp-server` or `iam-mcp-server`) come with a severe HTTP mode warning: 
> **"Single Customer Server... NOT designed for multi-tenant environments."**

### Why does this warning exist?
Standard `awslabs/mcp` servers initialize one **global** `boto3` SDK client upon startup by falling back to the host machine's default credentials (e.g., the EC2 Instance Profile). If multiple users hit this single HTTP server, the server will **ignore** the injected `x-aws-...` credentials and instead execute all actions using its own shared host role. This destroys tenant isolation and breaks the JIT credential propagation pipeline.

### Architectural Fixes for Target Routing
To safely route requests from the AgentCore Gateway to your MCP servers while preserving multi-tenant credentials, you must choose one of the following models:

| Hosting Architecture | How it Handles JIT Credentials | Pros | Cons |
|----------------------|--------------------------------|------|------|
| **1. Header-Parsing Pooled Server (Recommended)** | You deploy a custom or modified MCP server that explicitly extracts the `x-aws-access-key-id`, `x-aws-secret-access-key`, and `x-aws-session-token` headers (injected by our Interceptor) and builds a scoped `boto3` client natively *per request*. | Cost-effective. True multi-tenancy. Scales easily. | Requires writing custom code to intercept and use the HTTP headers. Cannot use vanilla `awslabs/mcp` servers without modification. |
| **2. Silo Model (Dedicated Containers)** | You provision a dedicated Fargate container running the vanilla `awslabs/mcp` server *per tenant*. You bypass the Lambda Interceptor JIT generation and instead assign the IAM role directly to the Fargate Task. | No code modification needed. Guaranteed isolation. | Expensive and difficult to orchestrate dynamically for every new user. |
| **3. Lambda Target Servers** | You run the MCP server code inside an AWS Lambda function target. Because Lambda invocations are isolated, you can securely inject the credentials directly into the invocation context or environment. | High isolation. Serverless scaling. | Requires packaging the Python/TypeScript MCP server into a Lambda deployment package. |

*Action Item for production: If you want to use the vanilla `aws-api-mcp-server`, you must fork the repository and modify the `__init__` or connection block to prioritize reading AWS credentials from the incoming HTTP request headers before falling back to the default credential provider chain.*
